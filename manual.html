Lag Watch

Displays the amount of lag currently present during gameplay. For those familiar with ScriptHawk, the conversion from Practice ROM lag watch to ScriptHawk lag watch is:

lagscripthawk = lagpracticerom - 2

Lag Watch

Displays the average lag over the past 16 frames

Position Watch

Displays the X, Y and Z co-ordinates of the player. Due to technical limitations, this rounds to the nearest integer for all 3 axis.

Floor Watch

Displays the Y value of the player's floor

Speed Watch

Displays the speed of the player, irrespective of lag.

Average Speed Watch

Displays the average speed of the player over the past 64 frames, irrespective of lag.

Stored Position Watches

Displays the co-ordinates indicated by the first and second set of stored positions

Phase Assistant Watches

Displays some feedback on phasewalk attempts.
The phasewalk assistant only

Timer Watch

Displays a timer which is controllable by the player, and also allows for automatic triggers for timer start and end. See Timer Settings for more on triggers.

Controls

  Start, Stop and reset the timer, irrespective of selected triggers.

Giant Kosha Timer Watch

Displays the timer responsible for controlling Giant Kosha's tantrums within Crystal Caves. For extensive information on how this works, look at this pastebin: Click Here.

The watch colors itself based on various factors about the timer.

Color Code

KOSHA TIMER: 800 Timer is frozen as the kong is within a certain subset of chunks (Chunk is the game's way of dividing up a map to reduce amount of map geometry to load)
KOSHA TIMER: 200 Timer is within a range where entering a Tag Barrel or Rocketbarrel will reset the timer back to 800
KOSHA TIMER: 100 Timer is within range of where the tantrum music will commence
KOSHA TIMER: 50 Timer is within range of where, if the Giant Kosha cutscene has not been viewed, the cutscene will be called up.

Movement State Watch

Displays the current movement state of the player as a hex value.

Angle Watch

Displays the angle of the player converted into degrees rounded to 2 decimal places and truncated to be within 0→360 degrees (in case of the player having the phase state.

If the player does have the phase state, the watch will be colored.

Color Code

ANGLE: 179.91 Kong has the phase state

Input Watch

Displays the inputs that are being read from controller 1 in a small graphic. The position of the watch and the stick threshold can be changed in the settings menu.

Held Actor Watch

Displays the actor index of the object that is currently being held by the player. A full list of what each actor index means can be seen here.

If you are performing grabbed item manipulation, then there are some extra pieces of information which can assist you with this. Since, with grabbed item manipulation, the actor may not exist, the actor index of your held actor is pointing to free memory and can be any value from 0→65,535.

If the pointer to the object you are holding matches an actor that is loaded into the actor array, the watch will be colored to signify this.

Color Code

HELD ACTOR: 98 The Kong's held actor pointer is pointing to an instance of an actor where that instance currently exists in the actor array.

ISG Timer Watch

Displays the timer responsible for controlling the game's Intro Story, mainly used in speedruns for a sequence break called "Intro Story Glitch"

Color Code

ISG TIME: 0:55.00 An ISG fadeout is pending
ISG TIME: 1:24.50 The game is fading into an Intro Story cutscene

Change Savestate

Changes which savestate will be referenced when saving and loading state. 4 savestate slots are currently supported.

Save State

Due to the limited amount of memory space available, the Practice ROM stores a limited amount of variables. The "Save State" function stores various game variables to the selected savestate object.

Stored variables

  • Inventory Base
    • Standard Ammo
    • Homing Ammo
    • Oranges
    • Crystals
    • Film
    • Health
    • Melons
  • Player Variables
    • X, Y and Z co-ordinates
    • Stored Damage
    • Character
    • Skew Angle
    • Facing Angle
    • Floor Height
  • Helm Timer Variables
    • Helm Start Time (Default: 10 minutes)
    • Helm Time passed
  • Intro Story Variables
    • Story Timer
    • Previous Fadeout Index
  • Kong Base
    • For each Kong:
      • Special move level
      • Simian slam level
      • Weapon Upgrades bitfield
      • Ammo Belt level
      • Instrument bitfield
      • Coin count
      • Instrument Energy
      • For each level:
        • Possessed Colored Bananas
        • Colored Bananas turned into Troff 'n' Scoff
        • Possessed Golden Bananas
  • Map Variables
    • Current Map
    • Last entrance
    • Nearest Tag Barrel Height
  • The permanent flag block
  • The temporary flag block
  • Does the savestate have saved data
    • If there is no saved data within the state, the state cannot be loaded
  • Savestate creation timestamp
  • RNG Value
A savestate cannot be made if the player is within a certain sublist of maps:

    Load State from position

    If the selected savestate has save data, this function will load the data from that savestate, reset the map and will cause a transition to the stored map. The kong will spawn in the position indicated by the position data inside the savestate.

    If the selected savestate has no save data, an error sound will play.

    If the map the player is in is within a certain sublist:

      Load State from exit

      If the selected savestate has save data, this function will load the data from that savestate, reset the map and will cause a transition to the stored map. The kong will spawn from the entrance you last entered from when the savestate was created as stored inside the savestate object.

      If the selected savestate has no save data, an error sound will play.

      If the map the player is in is within a certain sublist:

        View Savestate Variables

        Displays some information about the savestate to assist with identifying which savestate is which:
        • Has Data
          • Whether the savestate has data or not
        • Created
          • Displays how long since a savestate's creation
        • Map
          • Map index of where the savestate was created
        • Exit
          • Exit index of where the savestate was created
        • Character
          • Character index of the kong who the player was playing as when the savestate was created
        • X,Y,Z
          • X,Y and Z co-ordinates of the player when the savestate was created
        • Facing
          • The facing angle of the kong represented in degrees when the savestate was created
        • Skew
          • The skew angle of the kong represented in degrees when the savestate was created
        • Helm Timer
          • Whether the Helm Timer was on, and if so, what time
        • ISG Timer
          • Whether the Intro Story was active, and if so, what time

          Timer Start Trigger: L Press

          Timer will start when Player 1 presses .

          NOTE:  always starts the timer. This setting depicts that no other game variable will cause the starting of the timer.

          Timer Start Trigger: Map Load

          Timer will start when a map begins to load after all of the map files from ROM have been transferred to memory.

          Timer Start Trigger: Input

          Timer will start when the player presses a new button or moves the analog stick after having a neutral stick input.

          Pause Timer on Pause Menu

          Determines whether the timer will pause when the game's pause menu is open.

          Timer Finish Trigger: L Press

          Timer will finish when Player 1 presses .

          NOTE:  always finishes the timer. This setting depicts that no other game variable will cause the finishing of the timer.

          Timer Finish Trigger: Transition

          Timer will finish when a transition begins (eg. A "DK Spin" transition or a fade transition)

          Timer Finish Trigger: Cutscene

          Timer will finish when a cutscene is registered as playing

          Timer Finish Trigger: GB Collection

          Timer will finish when a kong is registered as touching a golden banana.

          Timer Finish Trigger: Transition End

          Timer will finish as soon as a transition is completed (eg. A "DK Spin" transition or a fade transition)

          Development: Building

          The Practice ROM repo allows any user to build a current development version of the DK64 Practice ROM. In order to build the Practice ROM, you will need the following:
          • A .z64 Donkey Kong 64 US Vanilla ROM
            • ROM Hash: CF806FF2603640A748FCA5026DED28802F1F4A50
            • The ROM must be titled "dk64.z64" and placed in the parent directory for the repository.
          • Python 3.x installed or a text editor which can run a python terminal (Eg. Sublime Text)
          • Lua for Windows 5.1.5, which can be found here: Link
          Once these are installed, run build.py in the Build directory. Assuming your vanilla ROM is in the right directory, everything should build as normal. The output Practice ROM will be in the Build directory titled "dk64-practice-rom.z64".

          If you have a problem with building this and have everything listed above, raise it as a GitHub issue.

          Development: Developing

          The Practice ROM is completely open source, and allows you to browse the code to see how it works. Additionally, you can develop additional code to add new features or edit existing ones.

          When you want to run the Practice ROM code in ScriptHawk, which is the main tool we use to run test ROM Hack code for DK64 ROM Hacks, you will need to do the following:
          1. Clone the ScriptHawk repository
          2. Run personaliseASM.py to modify the ASM in your local copy to your system's file structure
          3. Open BizHawk and run your vanilla DK64 ROM. Save an emulator savestate at the file select screen and open up the lua console. Run scripthawk.lua in the ScriptHawk Directory. After this, run the following command:
            • console.clear();loadASMPatch()
          4. After running this, select the Master.asm file in the Practice ROM directory
          To modify the source code of the Practice ROM, I would advise learning a few languages:
          • If you wish to modify the code that runs during gameplay (eg. Frame advance, Savestates etc.), you will need to learn MIPS Assembler. Looking at this manual and some time spent with Project64's debugger should assist you with learning this.
          • If you wish to make modifications to this website, you will need to learn HTML, Javascript and CSS.
          • If you wish to modify the games assets, you will need to learn some Python. Some of the game's assets have been listed on this excel spreadsheet regarding ROM location.
            Some assets may need to be decompressed when pulling from ROM. Whilst you can use Gedecompress to extract the assets, and does prove useful for determining file size, the Practice ROM requires that NOTHING in terms of copyrighted game assets are stored in the repository, and so the source code must use the GZip Python library.

          Development: Workflow

          If you wish to contribute to the Practice ROM, create a separate branch or a fork of the Practice ROM repository so you can make your updates without altering the main Practice ROM repository/branch.

          Once you are happy with the changes, make a pull request of your changes to the Practice ROM repository/branch. Before an update from an external contributor can be pushed to the main branch, a checklist of the update must be completed:
          • If the update includes a game code or python change:
            1. The update must work on an original N64 console and produce the intended effect
            2. The update must have been tested to a significant degree, including console testing, to ensure it is at least 99% bug free
            3. The update must provide something of substance and try to be somewhat space efficient (if additional ROM Code has been added)
            4. If a new Active Menu option has been added, it must be placed in a sensible spot
          • If the update includes a website change:
            1. It must work on the following browsers
              • Google Chrome
              • Mozilla Firefox
              • Microsoft Edge
            2. It must work with a variety of devices and screen resolutions
              • You can use Chrome Dev tools to mimic devices and screen resolutions
          • Irrespective of the purpose of the change:
            • Any new files must be placed in a sensible location. If your file is classed as a member of an item on the below list, place your file in that location
              • Site-Only non-HTML files: "./Site"
              • ROM Hack constants (eg. In-game addresses): "./Source/Memory"
              • ROM Hack code: "./Source/Features"
              • ROM Hack assets: "./Source/Non-Code"
              • Building scripts: "./Build"
